PLAN INTEGRAL DE IMPLEMENTACIÓN BACKEND, BASE DE DATOS, SINCRONIZACIÓN OFFLINE-FIRST Y OPERACIÓN ESCALABLE

PARTE 1: PRINCIPIOS RECTORES DEL SISTEMA

1. Arquitectura general
El sistema se implementará con:
- Frontend Angular.
- Backend NestJS con TypeScript.
- MySQL 8 como base de datos transaccional principal.
- Redis como soporte para colas, jobs y control operativo.
- Prisma como capa de acceso a datos.
- Almacenamiento de archivos desacoplado del motor SQL.

La arquitectura estará orientada a:
- Operación online y offline-first.
- Sincronización incremental confiable.
- Seguridad por roles, permisos y contexto operativo.
- Escalabilidad modular.
- Auditoría automática.
- Importaciones masivas controladas.
- Trazabilidad completa.
- Alto control sobre concurrencia y consistencia.

2. Convenciones globales
- Todas las entidades sincronizables usarán UUID v4 en VARCHAR(36).
- El frontend generará UUID v4 reales mediante crypto.randomUUID() o librería equivalente.
- Todas las fechas y horas persistidas se almacenarán en UTC.
- Toda la API será versionada bajo /api/v1/...
- Todas las tablas transaccionales y sincronizables tendrán como mínimo:
  - created_at
  - updated_at
  - deleted_at
  - row_version
- El borrado será lógico para toda entidad que participe en sincronización o auditoría operativa.
- El backend será la fuente oficial de verdad para reglas de negocio, conflictos, acumulados y KPIs.
- La respuesta de la API usará un sobre estándar.
- Los procesos pesados no se ejecutarán directamente en el request HTTP si superan el umbral operativo definido; se derivarán a cola.
- Cada request y job tendrá correlation_id para trazabilidad técnica.

3. Sobre estándar de respuesta
Formato general:

{
  "success": true,
  "data": {},
  "meta": {
    "page": 1,
    "pageSize": 50,
    "total": 1000,
    "nextCursor": null,
    "syncCursor": "2026-03-18T17:00:00.000Z|845122"
  },
  "error": null
}

4. Política de tiempo
- Todo valor persistido en base de datos se almacenará en UTC.
- El frontend convertirá a hora local solo para visualización.
- Toda fecha expuesta por la API se enviará en ISO 8601 con milisegundos.
- Los cursores de sincronización y paginación temporal serán determinísticos.

5. Política de borrado
- No se eliminarán físicamente registros operativos sincronizables.
- DELETE en API equivale a soft delete.
- El cliente recibirá cambios y eliminaciones mediante sincronización incremental.
- Solo tablas puramente técnicas de baja retención podrán tener purga física controlada según política de retención.

6. Política de consistencia de versiones
- Cada registro sincronizable tendrá row_version.
- Cada UPDATE o DELETE lógico incrementará row_version.
- updated_at se mantendrá para trazabilidad y filtros temporales.
- El control de conflicto se basará primariamente en row_version y secundariamente en updated_at.

7. Política de snapshots
En tablas operativas que dependan de otras entidades cambiantes, se guardarán snapshots descriptivos además de foreign keys para preservar contexto histórico.
Ejemplos:
- operator_name_snapshot
- work_order_number_snapshot
- machine_code_snapshot
- client_snapshot
- product_snapshot

8. Política de change tracking
Se mantendrá un registro unificado de cambios para sincronización incremental, auditoría técnica y diagnósticos. Esto permitirá desacoplar el pull incremental del escaneo intensivo de múltiples tablas.


PARTE 2: STACK TECNOLÓGICO

1. Backend
- Framework: NestJS
- Lenguaje: TypeScript
- ORM: Prisma
- Validación: class-validator + class-transformer
- Autenticación: Passport.js + JWT + Refresh Token
- Documentación: Swagger/OpenAPI
- Cola de procesos: BullMQ
- Logs técnicos: Pino o Winston
- Métricas y trazas: OpenTelemetry
- Rate limiting: middleware/guard dedicado

2. Base de datos
- MySQL 8
- Charset: utf8mb4
- Collation: utf8mb4_unicode_ci

3. Infraestructura complementaria
- Redis para colas, jobs, locks operativos y cache táctica
- Almacenamiento de archivos externo o local protegido
- Reverse proxy con TLS
- Entornos: local, dev, staging, producción

4. Archivos
- Almacenamiento recomendado: S3 compatible, Google Cloud Storage o almacenamiento local protegido por backend
- En base de datos solo se guardará metadata
- Los archivos tendrán hash, metadatos y validación de tipo real
- El acceso podrá ser mediante URL firmada o endpoint protegido

5. Exportaciones
- Excel: exceljs
- PDF: pdfmake o equivalente
- Las exportaciones grandes se generarán asíncronamente vía jobs

6. Pruebas
- Unitarias: Jest
- Integración: Jest + Testcontainers o BD de prueba dedicada
- E2E: Supertest
- Pruebas de carga: k6 o equivalente


PARTE 3: SEGURIDAD, AUTENTICACIÓN Y AUTORIZACIÓN

1. Modelo de acceso
Se utilizará un modelo RBAC con restricciones contextuales:
- Rol
- Permisos
- Estado activo del usuario
- Áreas asignadas
- Máquinas permitidas
- Perfil del dispositivo cuando aplique

2. Access Token
El access token incluirá:
- sub
- username
- role_code
- permissions
- active
- session_id
No incluirá el turno como claim fijo. El turno se calculará en backend según la configuración del sistema.

3. Refresh Token
- Rotación obligatoria en cada renovación.
- Persistencia hash del token.
- Revocación por sesión y por usuario.
- Soporte para logout por dispositivo y logout global.

4. Sesiones por dispositivo
Se registrarán sesiones explícitas por dispositivo:
- usuario
- tipo de dispositivo
- nombre o etiqueta del dispositivo
- IP
- último acceso
- estado
- token vigente/revocado

Esto permitirá:
- cerrar sesiones remotas
- detectar uso anómalo
- administrar tablets compartidas

5. Validación contextual
Ejemplos:
- Un operario solo podrá registrar producción en máquinas de su área.
- Un supervisor podrá revisar y corregir reportes dentro de su alcance.
- Un usuario inactivo no podrá autenticarse.
- Un dispositivo con perfil "impresión" podrá tener visibilidad limitada de sincronización y módulos.

6. Rate limiting y protección anti abuso
Se implementará:
- rate limit por IP
- rate limit por usuario
- rate limit estricto para login
- rate limit para refresh-token
- rate limit para sync push
- rate limit para importaciones

7. Auditoría
Se implementará un interceptor global para registrar automáticamente:
- CREATE
- UPDATE
- DELETE lógico
- LOGIN
- LOGOUT
- IMPORT
- EXPORT

Se registrará:
- usuario
- sesión
- entidad
- entity_id
- acción
- old_values
- new_values
- IP
- user agent
- correlation_id
- timestamp

La auditoría será append-only y no tendrá endpoints de modificación o eliminación.


PARTE 4: ESTRATEGIA OFFLINE-FIRST Y SINCRONIZACIÓN

1. Enfoque adoptado
La sincronización se implementará con:
- GET /api/v1/sync/pull
- POST /api/v1/sync/push

2. Cursor de sincronización
El sistema usará un cursor fuerte de sincronización compuesto por:
- changed_at
- change_id

Formato:
YYYY-MM-DDTHH:mm:ss.sssZ|123456

Este cursor:
- evita pérdida de cambios por empates de timestamp
- permite paginación incremental segura
- soporta procesamiento continuo de grandes volúmenes

3. Change log unificado
Se mantendrá una tabla change_log que registrará cada cambio relevante sobre entidades sincronizables. El sync pull leerá del change_log, no de todas las tablas operativas directamente.

Cada evento registrará:
- id
- entity
- entity_id
- operation
- row_version
- changed_at
- user_id
- correlation_id

4. Payload de mutación
Ejemplo:

{
  "clientId": "tablet-imp-01",
  "deviceProfile": "PRINT_STATION",
  "mutations": [
    {
      "mutationId": "b6f9b0f3-5e3f-4d38-94e1-318baf2b6b91",
      "entity": "print_reports",
      "entityId": "7a8f8f0b-d3d5-4d83-a939-21af3d9e52a1",
      "action": "CREATE",
      "baseRowVersion": null,
      "baseUpdatedAt": null,
      "clientTimestamp": "2026-03-18T11:30:00.000Z",
      "payload": {}
    }
  ]
}

5. Resolución de conflictos
La resolución se basará en:
- row_version del servidor
- baseRowVersion del cliente
- updated_at como respaldo informativo
- claves únicas de negocio

Reglas:
- CREATE: conflicto o duplicado lógico si viola PK o unique business key
- UPDATE: conflicto si row_version del servidor es mayor que baseRowVersion
- DELETE: conflicto si el registro cambió desde baseRowVersion
- mutaciones sobre registros eliminados lógicamente se evaluarán según regla de reactivación o rechazo del módulo

6. Idempotencia
Cada mutationId será único.
El backend registrará mutationId y resultado en sync_mutation_log.
Reintentos no duplicarán efectos.

7. Procesamiento del lote
- Se procesará en orden de llegada.
- Cada mutación tendrá transacción propia.
- El resultado del lote podrá ser parcial.
- El lote no fallará completo por una sola mutación inválida.

8. Pull incremental
GET /api/v1/sync/pull?cursor=2026-03-18T10:00:00.000Z|820100&scope=production,quality

El backend devolverá:
- cambios por entidad
- registros eliminados lógicamente
- siguiente cursor
- posibilidad de paginar si el volumen es grande

9. Alcance de sincronización
El sync podrá acotarse por:
- módulo
- perfil de dispositivo
- área
- permiso
- ventana temporal
- lotes máximos

Ejemplos de scope:
- catalogs
- inventory
- work_orders
- production
- quality

10. Estrategia para conectividad inestable
- Lotes configurables
- Reintentos exponenciales
- Confirmación parcial
- Payloads comprimibles
- Pull segmentado
- Límites por request para evitar saturar tablets o redes internas

11. Estado mínimo en frontend
- syncStatus
- pendingSyncCount
- syncCursor
- cola de mutaciones
- baseRowVersion por registro editado
- perfil del dispositivo
- alcance de sincronización habilitado


PARTE 5: IMPORTACIONES MASIVAS DESDE EXCEL

1. Estrategia general
El flujo de importación será:

Paso 1: subir archivo
Paso 2: registrar import_job
Paso 3: parsear archivo
Paso 4: almacenar filas en staging
Paso 5: validar estructura y reglas
Paso 6: generar preview
Paso 7: confirmar importación
Paso 8: ejecutar upsert por lote
Paso 9: registrar resumen final
Paso 10: auditar el proceso

2. Staging obligatorio
Toda importación relevante usará tablas de staging:
- work_order_import_rows
- clise_import_rows
- die_import_rows

Cada fila podrá tener estados:
- PENDING
- VALID
- INVALID
- APPLIED

3. Claves de negocio para upsert
- users: username
- areas: code
- permissions: code
- roles: code
- machines: code
- shifts: code
- clises: item_code
- dies: serie
- work_orders: ot_number
- stock_items: pallet_id cuando exista y aplique

4. Reglas de importación
- Las filas inválidas no detendrán automáticamente el job en modo tolerante.
- El backend devolverá errores detallados por fila.
- Las importaciones grandes se ejecutarán por cola.
- El usuario podrá consultar el estado del job y descargar resultados.

5. Implementación
Para upsert real masivo:
- SQL nativo con INSERT ... ON DUPLICATE KEY UPDATE
o
- staging + merge transaccional

No se usará createMany + skipDuplicates como sustituto de upsert.

6. Trazabilidad
Se guardará:
- usuario
- archivo
- hash del archivo
- entidad
- total filas
- válidas
- inválidas
- aplicadas
- job status
- correlation_id

CREATE TABLE areas (
    id VARCHAR(36) PRIMARY KEY,
    code VARCHAR(50) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL,
    description TEXT NULL,
    active BOOLEAN NOT NULL DEFAULT TRUE,
    row_version BIGINT NOT NULL DEFAULT 1,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at DATETIME(3) NULL
);

CREATE TABLE permissions (
    id VARCHAR(36) PRIMARY KEY,
    code VARCHAR(100) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL,
    description TEXT NULL,
    row_version BIGINT NOT NULL DEFAULT 1,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at DATETIME(3) NULL
);

CREATE TABLE roles (
    id VARCHAR(36) PRIMARY KEY,
    code VARCHAR(50) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL,
    description TEXT NULL,
    active BOOLEAN NOT NULL DEFAULT TRUE,
    row_version BIGINT NOT NULL DEFAULT 1,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at DATETIME(3) NULL
);

CREATE TABLE role_permissions (
    id VARCHAR(36) PRIMARY KEY,
    role_id VARCHAR(36) NOT NULL,
    permission_id VARCHAR(36) NOT NULL,
    row_version BIGINT NOT NULL DEFAULT 1,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at DATETIME(3) NULL,
    CONSTRAINT fk_role_permissions_role FOREIGN KEY (role_id) REFERENCES roles(id),
    CONSTRAINT fk_role_permissions_permission FOREIGN KEY (permission_id) REFERENCES permissions(id),

    id VARCHAR(36) PRIMARY KEY,

    name VARCHAR(100) NOT NULL,

    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at DATETIME(3) NULL
);

CREATE TABLE users (
    id VARCHAR(36) PRIMARY KEY,
    role_id VARCHAR(36) NOT NULL,
    name VARCHAR(100) NOT NULL,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    active BOOLEAN NOT NULL DEFAULT TRUE,
    last_login_at DATETIME(3) NULL,
    password_changed_at DATETIME(3) NULL,
    row_version BIGINT NOT NULL DEFAULT 1,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),

    deleted_at DATETIME(3) NULL,

);

CREATE TABLE user_assigned_areas (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    area_id VARCHAR(36) NOT NULL,

    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at DATETIME(3) NULL,
    CONSTRAINT fk_user_assigned_areas_user FOREIGN KEY (user_id) REFERENCES users(id),
    CONSTRAINT fk_user_assigned_areas_area FOREIGN KEY (area_id) REFERENCES areas(id),
    CONSTRAINT uq_user_area UNIQUE (user_id, area_id)
);

CREATE TABLE system_config (
    id VARCHAR(36) PRIMARY KEY,
    plant_name VARCHAR(150) NOT NULL,
    auto_logout_minutes INT NOT NULL DEFAULT 30,

    operator_message TEXT NULL,
    timezone_name VARCHAR(100) NOT NULL DEFAULT 'UTC',
    row_version BIGINT NOT NULL DEFAULT 1,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)
);

CREATE TABLE machines (
    id VARCHAR(36) PRIMARY KEY,
    area_id VARCHAR(36) NOT NULL,
    code VARCHAR(50) NOT NULL UNIQUE,
    name VARCHAR(100) NOT NULL,
    type VARCHAR(50) NOT NULL,
    status ENUM('OPERATIVE', 'MAINTENANCE', 'STOPPED', 'NO_OPERATOR') NOT NULL DEFAULT 'OPERATIVE',
    active BOOLEAN NOT NULL DEFAULT TRUE,
    row_version BIGINT NOT NULL DEFAULT 1,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at DATETIME(3) NULL,
    CONSTRAINT fk_machines_area FOREIGN KEY (area_id) REFERENCES areas(id)
);

CREATE TABLE refresh_tokens (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,

    token_hash VARCHAR(255) NOT NULL UNIQUE,
    expires_at DATETIME(3) NOT NULL,
    rotated_from_token_id VARCHAR(36) NULL,
    revoked_at DATETIME(3) NULL,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    CONSTRAINT fk_refresh_tokens_user FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE user_sessions (
    id VARCHAR(36) PRIMARY KEY,

    device_id VARCHAR(100) NULL,

    device_profile VARCHAR(50) NULL,
    ip_address VARCHAR(45) NULL,
    user_agent TEXT NULL,
    active BOOLEAN NOT NULL DEFAULT TRUE,
    last_seen_at DATETIME(3) NULL,
    revoked_at DATETIME(3) NULL,
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    CONSTRAINT fk_user_sessions_user FOREIGN KEY (user_id) REFERENCES users(id),
    INDEX idx_user_sessions_user (user_id),
    INDEX idx_user_sessions_active (active)
);

CREATE TABLE audit_logs (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(36) NULL,

CREATE TABLE change_log (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,

    entity_id VARCHAR(36) NOT NULL,

    user_id VARCHAR(36) NULL,
    correlation_id VARCHAR(100) NULL,

PARTE 8: REGLAS DE DOMINIO IMPORTANTES

1. Turno actual
El turno actual se calculará desde la tabla shifts y la configuración del sistema, no desde el JWT.

2. Permisos
Toda operación se evaluará en dos niveles:
- permiso funcional
- restricción contextual

3. Integridad histórica
Las tablas operativas conservarán snapshots contextuales.

4. Cálculo de KPIs
El backend calculará y consolidará KPIs.
Los dashboards consumirán datos oficiales del backend.

5. Metros acumulados y acumulados derivados
Toda actualización derivada se ejecutará de forma transaccional y auditada.

6. Relaciones hijas
Los reportes podrán recibirse con sus actividades embebidas en una sola operación transaccional.

7. Máquinas de estado
Entidades con flujo operativo tendrán estados controlados:
- work_orders
- incidents
- print_reports
- diecut_reports
Las transiciones inválidas deberán ser rechazadas por backend.

8. Locks operativos
Para registros sensibles se podrán usar locks temporales controlados para evitar edición concurrente manual.

9. Perfiles de dispositivo
El sistema podrá operar con perfiles como:
- PRINT_STATION
- DIECUT_STATION
- SUPERVISOR
- WAREHOUSE
Esto permitirá limitar:
- módulos disponibles
- datos sincronizados
- frecuencia de sincronización
- visibilidad de catálogos


PARTE 9: CONTRATOS DE SINCRONIZACIÓN

1. Pull
GET /api/v1/sync/pull?cursor=2026-03-18T10:00:00.000Z|820100&scope=production,quality&limit=500

Respuesta:

{
  "success": true,
  "data": {
    "changes": [
      {
        "entity": "work_orders",
        "entityId": "uuid-1",
        "operation": "UPDATE",
        "rowVersion": 5,
        "record": {}
      }
    ]
  },
  "meta": {
    "nextCursor": "2026-03-18T12:00:00.000Z|845122",
    "hasMore": false,
    "syncCursor": "2026-03-18T12:00:00.000Z|845122"
  },
  "error": null
}

2. Push
POST /api/v1/sync/push

{
  "clientId": "tablet-01",
  "deviceProfile": "PRINT_STATION",
  "mutations": [
    {
      "mutationId": "1c30cc3d-4ba1-4523-a8f6-51e62e2e54f4",
      "entity": "incidents",
      "entityId": "d1a1233e-20cf-44ae-9753-9d684f3d87db",
      "action": "CREATE",
      "baseRowVersion": null,
      "baseUpdatedAt": null,
      "clientTimestamp": "2026-03-18T11:30:00.000Z",
      "payload": {}
    }
  ]
}

Respuesta:

{
  "success": true,
  "data": {
    "processedMutationIds": [],
    "conflicts": [],
    "validationErrors": [],
    "serverChanges": []
  },
  "meta": {
    "syncCursor": "2026-03-18T12:00:00.000Z|845122"
  },
  "error": null
}

3. Criterios de conflicto
- UPDATE con baseRowVersion desfasado
- DELETE sobre registro modificado posteriormente
- CREATE con violación de unicidad
- mutación repetida ya aplicada
- mutación sobre registro eliminado en paralelo

4. Reintentos
- Un push puede reenviarse sin duplicar efectos gracias a mutationId
- El backend responderá en forma idempotente

5. Entidades incluidas
Inicialmente:
- machines
- work_orders
- clises
- dies
- stock_items
- incidents
- capa_actions
- print_reports
- diecut_reports
- tablas hijas requeridas
- catálogos mínimos necesarios para operación offline


PARTE 10: PAGINACIÓN Y CONSULTAS

PARTE 11: ARCHIVOS Y ENDURECIMIENTO

1. Política de archivos
- No almacenar binarios dentro de MySQL
- Validación de MIME real
- Límite de tamaño configurable
- Hash SHA-256
- object_key no predecible

- posibilidad de URL firmada

2. Escaneo
Se contemplará proceso de virus scan o validación de seguridad del archivo antes de habilitar acceso definitivo.

3. Relación con entidades
Los archivos estarán vinculados por:
- entity_name
- entity_id
