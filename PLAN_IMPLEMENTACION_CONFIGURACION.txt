# Plan de Ejecucion Implementable con Paradas Obligatorias

## Resumen
- La implementacion se hara exactamente por bloques criticos, con pausa obligatoria despues de cada uno para pruebas manuales antes de continuar.
- Se usaran las skills necesarias por fase:
  - `angular-developer`
  - `frontend-design` para cualquier cambio UI, preservando glassmorfismo y modo oscuro
  - `nodejs-backend-patterns`
  - `nodejs-best-practices`
  - `typescript-advanced-types`
- No se implementaran responsables por turno.
- Se reutilizara el flujo anfitrion actual del panel del operario.
- La seccion OT se implementara como modelo real, no como etiquetas visuales.
- Toda modificacion visual debe conservar el lenguaje actual del sistema:
  - modo oscuro
  - glassmorfismo
  - blur en superficies
  - bordes translucidos
  - profundidad visual consistente con admin, login y dashboard

## Reglas obligatorias de implementacion
- No avanzar al siguiente bloque sin cerrar el bloque actual y realizar su pausa de prueba.
- Toda fase debe usar las skills que correspondan a su capa tecnica.
- Si una fase toca frontend y backend, deben usarse ambas familias de skills.
- Si una fase modifica contratos o tipos compartidos, debe usarse tambien `typescript-advanced-types`.
- Si una fase modifica UI, aunque sea un ajuste menor, debe usarse `frontend-design` ademas de `angular-developer`.
- No se crearan tablas, DTOs, endpoints ni pantallas para responsables por turno.
- El estilo visual no debe desviarse del look actual dark/glass del producto.

### Skills a usar
- `nodejs-backend-patterns`
- `nodejs-best-practices`
- `typescript-advanced-types`
- `angular-developer`

### Backend
- Ampliar `system_config` con:

  - `maintenance_message`
  - `offline_retention_days`
  - `backup_frequency`
  - `conflict_resolution_policy`
  - `production_assistant_message`
  - `finishing_manager_message`
  - `management_message`
  - `failed_login_alert_mode`
  - `failed_login_max_attempts`
  - `ot_allow_partial_close`
  - `ot_allow_close_with_waste`

  - `ot_forced_close_requires_reason`

  - `auto_logout_minutes`

  - `password_policy_days`

## Fase 7. Migracion del modelo OT
### Objetivo
- Sustituir el estado OT actual por un modelo canónico nuevo y mas expresivo.

### Skills a usar
- `nodejs-backend-patterns`
- `nodejs-best-practices`
- `angular-developer`
- `typescript-advanced-types`

### Catalogo OT nuevo
- `POR_REVISAR`
- `EN_REVISION`
- `POR_ASIGNAR`
- `REVISADO`
- `PARCIAL`
- `ANULADO`
- `CERRADO`
- `POR_IMPRIMIR`
- `POR_FONDEAR`
- `POR_TROQUELAR`
- `POR_REBOBINAR`
- `POR_EMPAQUETAR`
- `FINALIZADO`

### Backend y base de datos
- Reemplazar `work_orders.status` por el nuevo catalogo.
- Agregar:
  - `workflow_route_json`
  - `workflow_step_index`
- Dejar de sobrescribir `Linea_produccion` desde planificacion.
- Migrar estados previos al modelo nuevo.

### Reglas de migracion
- `IMPORTED` o `PENDIENTE` -> `POR_REVISAR`
- `PLANNED` o `PLANIFICADO` -> `POR_ASIGNAR`
- `IN_PRODUCTION` o `EN PROCESO` -> primer `POR_*` resolvible
- `PARTIAL` o `PAUSADA` -> `PARCIAL`
- `COMPLETED` -> `FINALIZADO`
- `CANCELLED` -> `ANULADO`

### Pausa de prueba
- Importar o cargar OTs existentes.
- Confirmar migracion correcta.
- Confirmar preservacion de `Linea_produccion`.

## Fase 8. Avance automatico OT por reportes
### Objetivo
- Hacer que los reportes productivos muevan automaticamente la OT al siguiente estado esperado.

### Skills a usar
- `nodejs-backend-patterns`
- `nodejs-best-practices`
- `typescript-advanced-types`

### Regla general
- Reporte total o completo:
  - completa el proceso actual
  - avanza al siguiente `POR_*`
  - si no hay siguiente proceso, pasa a `FINALIZADO`
- Reporte parcial:
  - mueve la OT a `PARCIAL`

### Trabajo tecnico
- Centralizar la logica de transicion OT en una sola fuente de verdad.
- Hacer que impresion, troquelado, rebobinado y empaquetado usen esa logica.

### Pausa de prueba
- Registrar reportes reales o de prueba.
- Confirmar cambio automatico del estado OT.
- Confirmar secuencia correcta.

## Fase 10. Politica de cierre OT
### Objetivo
- Implementar el cierre OT como regla de negocio configurable, distinta de `FINALIZADO`.

### Skills a usar
- `nodejs-backend-patterns`
- `nodejs-best-practices`
- `angular-developer`
- `typescript-advanced-types`

### Configuracion
- `ot_allow_partial_close`
- `ot_allow_close_with_waste`
- `ot_allow_forced_close`
- `ot_forced_close_requires_reason`

### Reglas
- `FINALIZADO` significa fin del ultimo proceso productivo.
- `CERRADO` significa cierre explicito de negocio.
- Cierre desde `PARCIAL` solo si la politica lo permite.
- Cierre con merma solo si esta habilitado.
- Cierre forzado requiere flag y motivo cuando aplique.

### Pausa de prueba
- Probar cierre parcial.
- Probar cierre con merma.
- Probar cierre forzado con motivo.
