Implementación de Configuración como Contrato Real, con Respaldo Inicial, Pausas de Validación y Criterio de Código Limpio

Resumen
- La pestaña `Configuración` se tomará como contrato funcional y adaptaremos frontend, backend y base de datos a esa UI depurada.
- Antes de cualquier cambio de código, la primera acción de ejecución será crear un respaldo del plan en un archivo `.txt` en la raíz del proyecto.
- La implementación se hará por etapas secuenciales, y después de cada cambio mayor habrá una pausa obligatoria para validación manual antes de continuar.
- Se eliminan del contrato: `Áreas activas`, `Sincronización solo Wi‑Fi` y `2FA Obligatorio`.
- Decisiones ya cerradas:
  - `Modo Mantenimiento Global`: bloqueo total para todo usuario no `Sistemas`, mostrando mensaje de mantenimiento.
  - `Turnos`: exactamente 2 turnos configurables, con inicio y fin editables.
  - `Estados OT`: solo se editan etiquetas visibles, no la máquina de estados.
  - `Intentos fallidos`: registrar evento auditable y dejar backend listo para notificación in-app futura.
- Criterio de implementación:
  - mantener código limpio, cohesivo y predecible
  - evitar atajos o duplicaciones
  - no forzar cambios sobre modelos existentes si antes no se valida su impacto
  - preferir refactor pequeño y correcto sobre parche rápido

Principios obligatorios durante la implementación
- Analizar primero el flujo actual antes de tocar cada etapa.
- Cambiar solo la responsabilidad necesaria en cada capa:
  - frontend para bindings, estado y orquestación
  - backend para contratos, validación y reglas de negocio
  - base de datos para persistencia real
- No duplicar lógica entre frontend y backend cuando una sola capa pueda ser la fuente de verdad.
- No mezclar persistencia de `turnos` dentro de `system_config`; `shifts` seguirá siendo entidad separada.
- No hardcodear nuevas reglas si pueden modelarse de forma explícita y clara.
- Mantener nombres y tipos consistentes entre Prisma, DTOs, adapters, estado frontend y componentes.
- Si aparece una inconsistencia estructural durante una etapa, corregirla antes de seguir, aunque eso implique refinar el diseño de esa etapa.
- Cada etapa debe cerrar con código ordenado, sin deuda nueva evidente y con pruebas acordes al cambio.

Cambios en base de datos, APIs y tipos

Base de datos
- `system_config`:
  - mantener: `plant_name`, `auto_logout_minutes`, `password_expiry_warning_days`, `password_policy_days`, `operator_message`, `timezone_name`
  - agregar:
    - `maintenance_mode_enabled`
    - `maintenance_message`
    - `offline_retention_days`
    - `backup_frequency`
    - `conflict_resolution_policy`
    - `production_assistant_message`
    - `finishing_manager_message`
    - `management_message`
    - `failed_login_alert_mode`
    - `failed_login_max_attempts`
    - `ot_allow_partial_close`
    - `ot_allow_close_with_waste`
    - `ot_allow_forced_close`
    - `ot_forced_close_requires_reason`
    - `ot_label_created`
    - `ot_label_released`
    - `ot_label_in_process`
    - `ot_label_paused`
    - `ot_label_closed`
    - `ot_label_blocked`
- `shifts`:
  - mantener tabla actual
  - seguir usando `name`, `start_time`, `end_time`, `crosses_midnight`, `active`
- nueva tabla:
  - `shift_supervisor_assignments`
  - campos: `id`, `shift_id`, `user_id`, `role_scope`, `active`, `created_at`, `updated_at`

APIs
- ampliar `GET /system-config` y `PUT /system-config`
- mantener `GET /shifts` y `PUT /shifts/:id`
- agregar:
  - `GET /shift-supervisors`
  - `PUT /shift-supervisors/:shiftId`
- reutilizar `GET /audit/logs` para la tabla de auditoría
- agregar protección backend para mantenimiento global sobre rutas y mutaciones

Tipos frontend/backend
- ampliar `SystemConfig`
- crear `ShiftSupervisorAssignment`
- crear un view-model combinado para la pestaña Configuración
- extender el estado global para:
  - mensajes por rol
  - mantenimiento
  - etiquetas OT
  - política de cierre

Plan de pruebas por etapa
- Cada etapa debe cerrar con:
  - pruebas técnicas del módulo tocado
  - build si el cambio es transversal
  - revisión rápida de limpieza estructural del código modificado
  - validación manual tuya antes de continuar
- Validaciones mínimas acumuladas:
  - persistencia real de configuración
  - guardado coordinado sin éxito falso
  - mantenimiento global funcionando por rol
  - etiquetas OT reflejadas en UI
  - política de cierre respetada por backend
  - intentos fallidos auditados
  - auditoría real visible en la pestaña

Supuestos y defaults
- Los tres mensajes nuevos son globales por rol, no por turno.
- “Encargado de rebobinado y troquelado” se mapeará al grupo de roles operativos de finishing ya existentes.
- Las etiquetas OT son solo de presentación; el backend sigue usando estados internos fijos.
- La primera fase deja lista la base para notificación in-app por intentos fallidos, pero no obliga a implementar un centro de notificaciones completo.
- Skills a usar por defecto durante la implementación:
  - frontend: `angular-developer`
  - backend: `nodejs-backend-patterns` + `nodejs-best-practices`
  - contratos/tipos: `typescript-advanced-types`
