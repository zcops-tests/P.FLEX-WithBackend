# Plan Revisado con Micro-Pausas por Punto

## Resumen
- La UI actual queda congelada como contrato. No se modifica ninguna pantalla, layout, modal, tabla, tab, formulario o estilo, salvo que luego entregues una plantilla HTML explícita.
- La implementación se hará por fases, pero además **cada punto dentro de cada fase será un checkpoint obligatorio**.
- No se avanzará al siguiente punto sin tu validación. Tampoco se avanzará a la siguiente fase sin haber aprobado todos los puntos de la fase anterior.
- `reports/*` y `operator/*` siguen siendo el flujo productivo activo. `src/features/production/*` se considera legado y se elimina solo al final.
- `planning` se implementará como backend real, pero manteniendo intacta la UI actual de `Programación`.
- La `Ficha` será el eje técnico nuevo, sin reemplazar visualmente a la OT en la UI actual.

## Protocolo de ejecución
- Cada punto se implementa de forma aislada y cerrable.
- Al terminar cada punto se hará una pausa obligatoria y se entregará exactamente:
  - Qué quedó implementado.
  - Qué validaciones pasaron o fallaron.
  - Qué riesgos o deuda quedaron abiertos.
  - Pregunta de control: `¿Aprobamos este punto o ajustamos algo antes de seguir al siguiente?`
- Validación mínima por punto:
  - Frontend: `npm run lint` y `npm run build` si hubo cambios frontend.
  - Backend: `npm run build` y pruebas del módulo tocado si hubo cambios backend.
  - Smoke puntual del comportamiento intervenido.
- Regla de seguridad:
  - no se toca markup salvo necesidad técnica estricta de cableado;
  - si un punto requiere alterar la estructura visible de la UI, ese punto se detiene y se replantea antes de implementarse.

## Fase 1. Blindaje del estado actual
1. Levantar línea base de rutas, servicios activos y pantallas canónicas.
   - Confirmar oficialmente qué sigue vivo: `operator/*`, `reports/*`, `planning/*`, `admin/*`, `orders/*`.
   - Confirmar oficialmente qué queda legado: `src/features/production/*`.
2. Crear red de regresión para no romper lo que hoy funciona.
   - Smoke baseline de login anfitrión, identificación por DNI, impresión, troquelado, rebobinado, empaquetado, usuarios, máquinas y programación.
3. Mapear contratos actuales UI -> servicio -> backend.
   - Especial foco en `ScheduleComponent`, `OrdersService`, `ProductionService`, `AdminService`, `StateService`.
4. Eliminar dependencias lógicas no visibles que hoy son frágiles.
   - Operadores hardcodeados, fallbacks aleatorios, lecturas mock en caminos activos.
5. Dejar checklist de regresión obligatorio para cada siguiente punto.
   - Ningún punto posterior se considera cerrado si rompe un flujo baseline.

## Fase 2. Introducción de `Ficha` como nuevo eje técnico
1. Diseñar y crear entidad backend `Ficha`.
   - Debe soportar: número de ficha, descripción base, categoría oficial, observaciones, item de impresión cuando aplique y metadatos clave importados.
2. Relacionar `WorkOrder` con `Ficha` sin romper OT actual.
   - La OT sigue existiendo como unidad operativa y comercial.
3. Crear proceso de backfill e importación OT -> Ficha.
   - Se toma principalmente `Descripción` y `Nro. Ficha` desde Excel, más demás datos disponibles salvo cantidades productivas.
4. Definir regla de categoría oficial en ficha.
   - Categorías: `Etiqueta Flexo - Colores`, `Etiqueta - Sin Impresión`, `Etiqueta Fondeado - Colores`, `Etiqueta - Híbrida`, `Etiqueta - Zebra`, `Etiqueta Digital`.
   - Si no puede resolverse con seguridad: `UNCLASSIFIED`.
5. Exponer consultas de ficha y su relación con OTs.
   - La ficha debe poder devolver sus OTs asociadas sin duplicar datos.
6. Añadir historial técnico por ficha.
   - La ficha debe poder consultar OTs, reportes e incidencias relacionadas por vínculo, no por copia física.

## Fase 3. Planning backend real sin cambiar la UI actual
1. Crear `PlanningModule` backend.
   - Endpoints para crear, editar, listar y consultar asignaciones.
2. Modelar asignación de planning.
   - OT seleccionada, ficha resuelta, máquina, área, turno, fecha/hora de inicio, duración, operador responsable, notas y estado de validación.
3. Conectar `ScheduleComponent` a `PlanningService` real.
   - La pantalla visible debe seguir igual.
4. Mantener compatibilidad con el esquema actual de OT.
   - Al guardar planning también se refleja snapshot en `schedule*`, `fecha_programada_produccion` y estado planificado de la OT.
5. Reemplazar operadores hardcodeados del scheduler por usuarios reales.
   - Filtrados por área, rol y estado, sin cambiar el select actual.
6. Persistir conflictos reales.
   - Solapes por máquina, fecha, turno y horario deben detectarse en backend.
7. Persistir cambio de estado de máquina desde programación.
   - Manteniendo el selector visual actual.
8. Mantener tabs actuales de programación.
   - Solo `Impresión`, `Troquelado` y `Rebobinado`, sin cambios visuales.

## Fase 4. Validación técnica y compatibilidades por ficha
1. Crear relación propia `Ficha <-> Troquel compatible`.
   - Separada de `CliseDieLink`.
2. Reutilizar relaciones técnicas existentes para sugerencias.
   - `item_code`, `ficha_fler`, `troquel_ficha`, `acabado_ficha`, clisés y troqueles.
3. Integrar la validación de herramental del scheduler con backend real.
   - Los pasos visibles actuales del wizard se mantienen.
4. Guardar resultados de validación por asignación y por ficha.
   - Debe quedar historial trazable.
5. Registrar número de ítem en ficha cuando corresponda a impresión.
   - No se fuerza para otras áreas.
6. Hacer que la lista de troqueles compatibles crezca desde validación.
   - Sin introducir una UI nueva.
7. Bloquear cierre de validación si la ficha no tiene categoría oficial o reglas mínimas.
   - La UI actual se conserva; cambia solo el comportamiento.

## Fase 5. Integración de producción real con ficha
1. Vincular reportes de impresión con ficha a través de OT.
2. Vincular reportes de troquelado con ficha a través de OT.
3. Vincular reportes de rebobinado con ficha a través de OT.
4. Vincular reportes de empaquetado con ficha a través de OT.
5. Exponer historial por ficha para reportes e incidencias.
   - La OT sigue mostrando solo su propio historial particular.
6. Estabilizar adapters frontend para que `reports/*` consuma modelos normalizados.
   - Sin cambiar la UI visible actual.
7. Confirmar que `operator/*` y `reports/*` siguen siendo el flujo activo único.
   - Sin dependencia funcional de `src/features/production/*`.

## Fase 6. Administración y configuración sin rediseño
1. Completar backend real de `SystemConfig` para soportar la pantalla actual.
   - Mantener los controles actuales, pero con persistencia real.
2. Soportar turnos, mensajes al operador y políticas de sesión/contraseña.
3. Soportar mantenimiento global y reglas de seguridad visibles.
4. Soportar persistencia de parámetros de sync/backup/conflictos que hoy aparecen en la UI.
   - Aunque algunos efectos operativos complejos puedan activarse en fases posteriores, la configuración debe dejar de ser decorativa.
5. Endurecer `AdminService` y `StateService` para que configuración, usuarios, roles y máquinas no dependan de supuestos implícitos.
6. Sustituir `alert()` y patrones legacy por el sistema actual de notificaciones.
   - Sin rediseñar pantallas.

## Fase 7. Limpieza final y retiro de legado
1. Confirmar que ninguna ruta activa depende de `src/features/production/*`.
2. Migrar cualquier lógica útil restante de `src/features/production/*` al flujo canónico si aún existiera.
3. Eliminar físicamente `src/features/production/*`.
4. Quitar señales falsas del layout existente.
   - Badge fijo de OTs, latencia aleatoria y estados simulados.
5. Verificar que dashboard y sidebar usen datos reales detrás de la UI actual.
6. Mantener `InventoryInk` fuera del alcance funcional de esta ola.
   - No se completa ni se rediseña sin plantilla explícita.

## Cambios de APIs, interfaces y tipos
- Backend:
  - Nueva entidad `Ficha`.
  - Relación `Ficha -> WorkOrder`.
  - Nueva relación `Ficha -> Troqueles compatibles`.
  - Nuevo `PlanningModule` con asignaciones y validaciones.
  - Extensión de `SystemConfig` para cubrir la UI actual.
- Frontend:
  - `ScheduleComponent` conserva UI y cambia de fuente de datos a `PlanningService`.
  - `OrdersService` mantiene compatibilidad con agenda snapshot de OT.
  - `ProductionService` y modelos de reportes se consolidan como contrato activo.
  - No se crea una UI nueva para ficha; ficha vive detrás de las pantallas existentes.

## Pruebas y aceptación
- Baseline permanente:
  - login anfitrión
  - identificación de operario por DNI
  - impresión
  - troquelado
  - rebobinado
  - empaquetado
  - gestión de usuarios
  - gestión de máquinas
  - programación
- Aceptación funcional:
  - La UI debe verse igual antes y después de cada punto.
  - La OT sigue siendo visible en programación y reportes.
  - La ficha aparece como eje de backend y de validación, no como cambio visual forzado.
  - Ningún punto puede cerrar si rompe flujos ya aprobados.

## Supuestos y defaults
- “Punto” significa cada ítem numerado de las fases anteriores.
- No se tocará la UI salvo que exista una necesidad mínima de cableado técnico sin cambio visual o una plantilla HTML explícita entregada por ti.
- La ficha será la fuente oficial de categoría, observaciones técnicas, historial transversal y compatibilidades.
- La OT seguirá siendo la fuente oficial de cantidades y del historial particular de esa OT.
- `Programación` seguirá usando la UI actual, pero por detrás dejará de depender principalmente de campos `schedule*` embebidos en la OT.
- `Packaging` mantiene su UI actual.
- `InventoryInk` queda fuera de esta ola hasta tener definición visual y funcional explícita.
